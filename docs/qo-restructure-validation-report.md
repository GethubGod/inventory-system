# Quick Order Restructure — Independent Validation Report

## 1. Summary

Codex shipped the qo_* restructure (5 new tables, 1 new column on current_stock_snapshots, sync rewrite, edge-function rewrite, 20-test contract). Independent validation found Codex's typecheck/lint/jest claims hold (645 tests pass, 14 intentionally skipped for retired safety/cap behavior; tsc and eslint clean). However, adversarial review surfaced 14 substantive bugs spanning critical (4), high (6), and medium (4) severity. 7 were fixed in this pass via 1 supplemental migration and direct edits to the parse-order edge function and Google Sheets sync; 7 remain documented but unfixed (some intentionally deferred per spec, some out of safe scope). Production-readiness verdict: **YELLOW** — safe to deploy the schema migration after the supplemental fix-up migration is also applied, but the operator must manually re-create employee custom counting unit rows (e.g. Nate's Tamago "order") on the new `personalization` sheet because the legacy data lived in `item_allowed_units` which the spec intentionally discards.

## 5. Remaining Risks

- Custom counting unit semantics depend on current_stock_snapshots upsert resolving to the right row. With the supplemental migration's full (non-partial) unique index on (entered_by_user_id, item_id, location_id, tracking_unit_key), concurrent re-counts for the same employee + item + tracking_unit will land last-write-wins; this is the design but could surprise an operator if two people count simultaneously under one employee account.
- Holiday overrides are synced but not consumed at runtime. A planted operator entry will silently no-op until the recommendation engine wires it up.
- Per-employee custom counting unit data lives in item_allowed_units which the spec discards. After migration the operator must re-enter Nate's "order" units for Tamago/Squid/Wakame/Chili Oil/etc on the new `personalization` sheet (tab=`personalization`, rule_type=item_config, personal_unit=order, personal_unit_equals blank, trigger_at_or_below=…, order_qty=1, order_unit=pack). If they forget, Nate's parser experience degrades silently to global behavior.
- Recommendation engine's tracking_unit blindness (Issue 11) means a single employee re-counting in a different unit than they normally use can produce a recommendation in the wrong unit-of-measure. Risk grows once Nate's custom-unit data is populated.
- Sync polymorphic validation rejects rows individually but a row with truthy `false` style cells (e.g., spreadsheet booleans rendered as the string "FALSE") routes through normalizeOptionalBoolean — verify the operator sheet's data validation dropdowns are set up so the column types are consistent.
- The 14 skipped jest tests cover behavior that has been retired (safety caps, allowed-units enforcement, per-employee unit rule v1). If the team ever re-introduces any of those behaviors, the tests must be re-enabled or the regressions will not be caught.
